# Research: CO Entity Certified Document Download

**Feature**: 034-co-entity-doc-download  
**Date**: 2026-04-08

## R-001: CO SOS Certified Copies Navigation Flow

### Decision: 3-step navigation with href extraction from certificate print page

### Rationale
Live exploration of the CO SOS website (entity: SCHOLER WENDYS COLORADO LLC, ID# 20201222644) revealed that clicking "Get certified copies of documents" navigates through three distinct pages:

1. **Entity Details** (`BusinessEntityDetail.do`) — link text: "Get certified copies of documents", selector: `a:has-text('Get certified copies of documents')`
2. **Certified Copies Selection** (`CertificationCopiesSelect.do`) — shows document list table + "All documents" link, selector: `a:has-text('All documents')`
3. **Certificate Print** (`CertificationPrint.do`) — embedded PDF viewer with "open your form in a new window" link containing the actual PDF URL (`CertificationViewer.do`)

The PDF is NOT served as a direct download. The "All documents" link navigates to a page with an embedded viewer; the actual PDF is accessible via the dynamically-generated `CertificationViewer.do?masterFileId={id}&certId={dynamicId}` URL found in the "open your form in a new window" href.

### Alternatives Considered
- **Direct URL construction**: Rejected because `certId` is dynamically generated by the server per request and cannot be predicted.
- **Intercepting embedded PDF viewer**: More complex, browser-engine-dependent. The href extraction is simpler and reliable.

## R-002: PDF Download Strategy — Browser API Request Context

### Decision: Extract `CertificationViewer.do` href and fetch via `browserContext.APIRequest.GetAsync()`

### Rationale
The CertificationViewer.do endpoint requires session cookies from the prior navigation flow (entity details → certified copies → certificate print). The browser API request context (`browserContext.APIRequest`) automatically carries session cookies, matching the pattern used for WY history document downloads in `ProcessHistoryDocumentsAsync`.

Implementation approach:
1. On the Certificate Print page, extract the href from `a:has-text('open your form in a new window')`
2. Resolve it to an absolute URL
3. Use `browserContext.APIRequest.GetAsync(absoluteUrl)` to download the PDF
4. Validate content-type contains `application/pdf` and body length > 0
5. Save locally to diagnostics folder as `{idNumber}-alldocuments.pdf`
6. Upload to blob storage under `{blobPrefix}/{filename}`
7. Build proxy URL and store in the `certifiedDocumentsLocalPath` variable

### Alternatives Considered
- **`page.WaitForEvent("download")` with link click**: Works for direct downloads but the "open your form in a new window" link opens an inline PDF, not a download event. Rejected.
- **Navigate to CertificationViewer.do and capture page content**: Over-complex; API request is simpler and avoids page navigation side effects.

## R-003: JSON Definition Action Sequence

### Decision: Append actions 005–008 after existing certificate download (action-004)

### Rationale
The existing `us-co-entity-details.json` has actions 000–004. The certificate of good standing download (action-004) uses `page.WaitForEvent("download")` which does NOT navigate away from the entity details page. Therefore, new actions can be appended sequentially:

| Action | Type | Description |
|--------|------|-------------|
| action-005 | click | Click "Get certified copies of documents" link (conditional: element-exists) |
| action-006 | screenshot | Screenshot the certified copies list page |
| action-007 | click | Click "All documents" link, wait for navigation |
| action-008 | extract | Extract the "open your form in a new window" href |
| (post-process) | download-via-api | Fetch PDF from extracted URL, upload to blob, store proxy URL |

**Note on action-008 and download**: The JSON definition engine's existing `download` action type uses `page.WaitForEvent("download")` with a click. This won't work here because the PDF is embedded, not downloaded. Instead, the download must be handled as CO-specific post-processing in `GenericScriptScraper` (similar to `ProcessMoFilingDocumentsAsync` and `ProcessWaFilingDocumentsAsync`), or add a new `extract` action that extracts the href, then use `ProcessHistoryDocumentsAsync`-style blob upload in post-processing.

**Chosen approach**: Use `extract` action (action-008) to get the CertificationViewer.do URL from the page, store it as `certifiedDocumentsViewerUrl`, then add CO-specific post-processing in `GenericScriptScraper.ExecuteEntityDetailScriptAsync` to download the PDF via `browserContext.APIRequest.GetAsync()`, upload to blob, and replace the URL with a proxy URL stored as `certifiedDocumentsLocalPath`.

## R-004: Response Integration — `certifiedDocumentsUrl` Field

### Decision: Flat sibling field in output compose block, mapped by EntityDetailMapper

### Rationale
Per clarification, `certifiedDocumentsUrl` is added as a flat sibling to `certificateUrl` in the JSON definition's output compose block:

```json
"compose": {
  "details": "details",
  "registeredAgent": "registeredAgent",
  "certificateUrl": "certificateLocalPath",
  "certifiedDocumentsUrl": "certifiedDocumentsLocalPath"
}
```

`EntityDetailMapper.MapCoDetails` maps this to the unified response. The value is either a proxy URL (on success), null (when link not present), or an error object (when download fails).

### Alternatives Considered
- **HistoryDocumentResult pattern**: Over-complex for a single combined document. Rejected.
- **Nested documents object**: More structural change than needed. Rejected per clarification.

## R-005: Test UI Integration

### Decision: Add "Certified Documents" download button in CO entity details view

### Rationale
The test UI already has download buttons for certificates and filing history documents. Adding a "Certified Documents" button follows the same pattern — check if `certifiedDocumentsUrl` is present in the response, render a button that fetches the document via the authenticated proxy URL using `URL.createObjectURL` (same as feature 028 document proxy auth pattern).
